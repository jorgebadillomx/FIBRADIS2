## Qué Salió Bien

1. **La decisión de cancelar 4-5-2 fue la correcta**: en lugar de scraping del cuerpo completo (legal ambiguo, frágil, alto mantenimiento), el módulo usa resumen Gemini + og:image + link externo. El resultado es mejor experiencia de lectura y zero problemas legales.

2. **SSRF protection en OgImageScraper sólida desde el inicio**: `OgImageScraper` implementa validación de hostname contra IP privadas, protocolo HTTPS forzado, Range header para solo descargar los primeros bytes (no el archivo completo), y timeout estricto. La seguridad fue iterada en varias pasadas de review y quedó en un estado defensivo correcto.

3. **`AllowAutoRedirect = true` en el cliente HTTP de og:image**: la mayoría de URLs de og:image en Google News son redirects (302/301). El fix en 4-6 fue un hallazgo real de uso — sin él, el 80% de imágenes no se cargaban. El review de 4-6 era limpio excepto por este y el Gemini null log.

4. **HTML snippet stripping en GoogleNewsRssClient**: el feed de Google News devuelve el snippet con HTML embebido (`<b>`, `<br>`, entidades HTML). El strip de HTML en el cliente fue un hallazgo de datos reales — no detectable con mocks.

5. **`AiMode.Manual` → `AiMode.On` rename**: la renombración fue quirúrgica y coherente en todo el stack (dominio, BD, API, Ops SPA, Public SPA). El rename mejoró la claridad semántica del modo: "On" comunica mejor el estado que "Manual".

6. **`NoticiaPage` (4-5-3) con SSR metadata correcta**: el patrón de metadata SSR (título, descripción, og:image) fue el punto de mayor atención en review. La implementación final con `fetchArticleById` tipado y metadata reactiva a la respuesta es limpia.

7. **Home muestra 5 noticias en lugar de 3**: la corrección de límite fue menor técnicamente pero significativa en UX — la sección de noticias con solo 3 artículos se veía escasa en pantallas grandes.

## Dificultades y Patrones Detectados

### 1. og:image scraping requirió múltiples iteraciones de seguridad

La historia 4-5-1 tuvo el review de seguridad más intensivo del proyecto:

- Pass 1: SSRF básico (rechazar IPs privadas por string)
- Pass 2: SSRF por DNS (hostname puede resolver a IP privada después de pasar el check inicial) → añadir resolución DNS + validación de la IP resultante
- Pass 3: protocol-relative URLs (`//cdn.example.com/img.jpg`) bypass HTTPS check → normalizar antes de validar
- Passes posteriores: detalles de OCE (OperationCanceledException), IPv4-mapped IPv6 (`::ffff:192.168.1.1`), skeleton state en UI, SocketException logging

Conclusión: el scraping de contenido externo es inherentemente un espacio de seguridad no trivial. Estos problemas son esperados y el proceso de review iterativo los atrapó todos antes de producción.

### 2. Datos reales revelaron bugs que los mocks ocultaron (patrón confirmado)

Este patrón ya documentado en la retro original se confirmó en 4-5-1 y 4-6:

- **og:image redirects**: ningún mock de HTTP simula redirects 302 por default
- **HTML en RSS snippet**: los mocks de RSS usaban strings planas, no HTML real
- **Gemini null en casos edge**: el mock de Gemini siempre retornaba una respuesta; el real puede retornar null en rate limit / error de parsing

Los tres eran bugs reales que solo el uso con datos verdaderos reveló.

### 3. Gemini null log (hallazgo de review 4-6)

El servicio `GeminiAiSummaryService` no loggeaba cuando retornaba null (rate limit, error de parsing, respuesta vacía). Desde el exterior, el silencio era indistinguible de "artículo procesado sin AI summary". El fix añadió un `LogWarning` en el path null. Hallazgo menor pero de alta visibilidad operativa.

### 4. Acumulación de deuda técnica Ops SPA

Los helpers de auth en Ops SPA siguen duplicados (`newsApi.ts`, `aiModeApi.ts`, etc.). No bloqueó ninguna historia de la Épica 4 extendida, pero la deuda crece con cada nueva historia de Ops. Necesitará consolidación antes de que Ops SPA añada gestión de sesión compleja.

## Aprendizajes Clave

1. **La cancelación de una historia es una decisión de producto válida y debe documentarse.** La historia 4-5-2 (scraping cuerpo completo) fue cancelada con una razón clara: el beneficio no justificaba los riesgos legales y técnicos. El tracker refleja el estado `cancelled` explícitamente.

2. **El scraping de imágenes externas requiere un modelo de amenaza explícito en el spec.** SSRF, protocol-relative URLs, DNS rebinding, IPv4-mapped IPv6 — estos no son casos edge rarísimos; son la realidad del internet. El Dev Notes de 4-5-1 los enumeraba; el review verificó que todos fueran atendidos.

3. **Los bugs de datos reales (HTML en RSS, og:image redirects, Gemini null) confirman el value del smoke test obligatorio en historias con integraciones externas.** Esta práctica se estableció como aprendizaje en la retro original y se validó en la práctica en estas historias.

4. **El rename de `AiMode.Manual` → `AiMode.On` muestra que los nombres de dominio deben revisarse antes de persistir.** Un nombre que tiene sentido en el contexto de "manual vs off" puede no tener sentido cuando el estado evoluciona a "on vs off". El costo del rename fue bajo porque el codebase es pequeño; en proyectos más grandes habría sido mayor.

5. **Los límites de contenido en UI (3 vs 5 noticias) deben estar en el spec, no hardcodeados como constantes mágicas sin referencia al diseño.** El cambio de 3 a 5 noticias en Home fue una corrección de diseño — no de código. El código en sí era correcto; el número era el problema.
