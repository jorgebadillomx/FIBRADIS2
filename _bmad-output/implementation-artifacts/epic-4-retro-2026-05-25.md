## Narrativa: Evolución del Pipeline de Extracción de Contenido

La Épica 4 extendida siguió una trayectoria de tres fases para la extracción del cuerpo de los artículos:

**Fase 1 — Sin parser (4-5-4):** La primera aproximación fue semántica sin dependencias externas. El algoritmo: primero buscar elementos `<article>`, `itemprop="articleBody"`, `<main>`; si no hay, extraer `<p>` con ≥60 chars usando regex. Calidad gate: ≥200 chars y sin el sentinel "Google News". Funcionó para casos simples. 78 tests unitarios y 6 de integración.

**Fase 2 — Control editorial (4-7-editor):** En paralelo con la mejora técnica, se añadió la capacidad de editar manualmente el body_text desde Ops. Esto reconoció que la extracción automática no puede ser perfecta: hay sitios con estructuras DOM inusuales, paywalls parciales, o contenido embebido que el algoritmo no puede resolver. El editor permite que el operador corrija casos edge sin tocar el código. 95 tests.

**Fase 3 — DOM parsing real (4-8):** La limitación del regex se hizo evidente en artículos con `<article>` anidados (el artículo interno era un widget de publicidad) y en sitios con estructuras DOM no estándar. AngleSharp v1.4.0 resuelve el problema de raíz con un parser DOM real. El `SiteExtractionCatalog` con 22 dominios (11 sitios financieros mexicanos + 3 RE especializados + 8 internacionales) permite estrategias de extracción por dominio. 84 unit + 6 integration tests.

**Conclusión de la evolución:** La secuencia 4-5-4 → 4-8 muestra que el pragmatismo inicial ("no añadir dependencias") es correcto para validar el approach, pero que eventualmente el DOM parsing real es necesario para calidad de producción. Añadir AngleSharp no fue una señal de que 4-5-4 fue un error — fue una señal de que el problema requería más potencia que regex.

## Qué Salió Bien (Historias 4-5-4, 4-7-editor, 4-8, 4-9)

### 1. Multi-proveedor AI con `RoutingAiSummaryService` (4-9)

La arquitectura es limpia: `RoutingAiSummaryService` delega a la implementación activa leyendo `AiProviderConfig` desde BD. Añadir un nuevo proveedor requiere únicamente implementar `IAiSummaryService` y añadir una entrada al enum. La historia 4-9 implementó Gemini (existente) y DeepSeek (nuevo); el framework soporta agregar más sin cambiar el routing. Fue la historia con menos patches de las cuatro (~5 patches, la más baja).

### 2. Editor manual como complemento a la extracción automática (4-7-editor)

Reconocer que la automatización no puede ser perfecta y añadir control humano es más valioso que perseguir el 100% de precisión en el algoritmo. El patrón `bodyTextPreview` (primeros 200 chars en el listado) + carga completa `GET /api/v1/ops/news/{id}` al abrir el editor es eficiente: no se transmite el cuerpo completo en el listado paginado. La limpieza (set to null) permite resetear un body_text corrupto. 95 tests validando paginación, edición y limpieza.

### 3. `SiteExtractionCatalog` como configuración de extracción por dominio (4-8)

El catálogo de 22 dominios es la pieza que convierte AngleSharp de "parser genérico" a "estrategia por sitio". Cada entrada del catálogo define el selector CSS preferido para ese dominio. Cuando un artículo no está en el catálogo, el pipeline cae al selector genérico (semántico). Esto significa que el mantenimiento futuro para un sitio problemático es añadir una línea al catálogo, no cambiar el algoritmo.

### 4. DeepSeek como proveedor alternativo real (4-9)

La API de DeepSeek es compatible con OpenAI (mismo SDK, distinto endpoint), lo que simplificó `DeepSeekAiSummaryService` a una reimplementación trivial de `GeminiAiSummaryService` con el cliente correcto. El seed por default es Gemini con `gemini-2.5-flash`; DeepSeek está disponible desde Ops sin redespliegue. 190 tests (la mayor cobertura de la Épica 4).

## Dificultades y Patrones Detectados (Historias 4-5-4, 4-7-editor, 4-8, 4-9)

### 1. Dos iteraciones para resolver la extracción de body_text

La secuencia 4-5-4 → 4-8 requirió dos historias para llegar al estado de producción. La primera iteración (regex) era suficiente para el MVP pero no para el dominio completo. Esto no es un problema de planificación — el problema de DOM parsing no es predecible hasta ver datos reales — pero sí es un patrón: las historias de extracción de contenido web requieren más iteraciones que las de dominio limpio.

### 2. `4-7-seleccion-modelo-ia-noticias` cancelada y absorbida por 4-9

La historia de selección de modelo IA (Flash vs Pro) fue cancelada porque 4-9 la absorbió con una arquitectura más completa (multi-proveedor en lugar de multi-modelo de un proveedor). La decisión fue correcta: el selector de modelo quedó como campo en el `AiProviderConfig` de 4-9. El costo fue el work-in-progress de la historia cancelada, aunque fue mínimo porque se detectó antes de implementar.

### 3. Caso `<article>` anidado (bug que motivó 4-8)

El bug que motivó AngleSharp era específico: algunas páginas tienen `<article class="ad-widget">` dentro del `<article>` principal. La estrategia de "tomar el primer `<article>`" falla en este caso. La solución de AngleSharp combina profundidad de árbol y longitud de texto para elegir el nodo correcto. Este tipo de bug es invisible en datos sintéticos y solo aparece con el corpus real de artículos de Google News.

Dana (QA Engineer): "Si hubiéramos testeado con artículos reales desde 4-5-4, habríamos encontrado el bug antes. Pero hay 22 dominios en el catálogo — no es práctico testear todos en la primera iteración."

## Aprendizajes Clave (Histórias 4-5-4, 4-7-editor, 4-8, 4-9)

1. **El primer approach sin dependencias externas es válido para explorar, pero no es el destino final en parsing de DOM web.** 4-5-4 entregó valor y estableció los criterios de calidad (quality gate, sentinel check); 4-8 llevó el resultado a producción. El orden importa: sin 4-5-4, 4-8 habría sido más difícil de especificar.

2. **El `SiteExtractionCatalog` es el patrón correcto para extracción por dominio.** Alternativa: lógica condicional en el servicio. El catálogo es más fácil de mantener, testear y extender. Cada nuevo sitio problemático es una entrada, no un if.

3. **La arquitectura multi-proveedor (`RoutingAiSummaryService`) es preferible a multi-configuración de un proveedor.** Añadir un proveedor nuevo es implementar una interfaz. Añadir una opción nueva a un solo proveedor es modificar código existente. La primera es open/closed; la segunda no.

4. **El control humano (editor manual) es parte del pipeline de calidad, no un fallback de última instancia.** Para contenido web heterogéneo, hay casos que ningún algoritmo resuelve bien. El editor de 4-7 es una válvula de escape que mejora la calidad sin bloquear el pipeline automático.

5. **Cancelar una historia a tiempo es correcto.** `4-7-seleccion-modelo-ia-noticias` fue cancelada antes de implementar al descubrir que 4-9 cubría el mismo objetivo con mejor arquitectura. El costo fue mínimo; el beneficio fue evitar deuda duplicada.
