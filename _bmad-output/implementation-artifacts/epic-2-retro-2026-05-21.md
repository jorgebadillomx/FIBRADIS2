## Dificultades y Patrones Detectados

### 1. Refinamiento post-cierre de historia: 2-5 → 2-6

La historia 2-6 fue creada después de la retrospectiva original como mejora del trabajo de 2-5. Esto genera un patrón que se repite en FIBRADIS: las historias de UI tienden a necesitar una segunda iteración de reorganización una vez que se tiene el primer resultado funcional visible.

No es un problema de arquitectura, sino de expectativas. Las historias de presentación evolucionan más que las de backend porque el criterio de "buena UI" es más subjetivo.

### 2. ARIA roles diferidos sistemáticamente

La tabla de universo de FIBRAs tiene una deuda explícita de `role="row"` / `role="rowheader"` / `role="gridcell"` diferidos por complejidad de markup. Este es el segundo o tercer componente de tabla en la épica con accesibilidad ARIA incompleta. No es un blocker funcional, pero el patrón de diferir accesibilidad en tablas complejas se está consolidando como hábito.

### 3. Cierre de historias inconsistente (problema previo no resuelto)

La inconsistencia reportada en la retrospectiva original (story 2-2 marcada `review` en su archivo pero `done` en sprint-status) no fue normalizada. Los action items de retro sin mecanismo de seguimiento tienden a no ejecutarse — patrón ya documentado en la Épica 3.

## Aprendizajes Clave

1. **Las historias de UI de Home/lista requieren presupuestar una iteración de reorganización.** La historia 2-6 no fue deuda técnica — fue refinamiento esperado. La primera versión funcional (`TopMovers` como tabla separada) fue el scaffolding correcto; la segunda versión (`GainersLosers` en el grid principal + `FibraUniverseTable`) fue el layout final. Ambas eran necesarias.

2. **Las tablas complejas tienen un costo de accesibilidad no trivial.** Si los criterios de aceptación no incluyen explícitamente los roles ARIA, se diferirán. Considerar añadirlos a la definición de done para componentes de tabla en historias futuras.

3. **El code review agrega valor incluso en historias de frontend "simples"**: los 2 patches de 2-6 (alineación de columnas, isError handling) eran bugs reales que habrían llegado a producción.

4. **`sprint-status.yaml` como fuente única de verdad funciona bien para tracking general**, pero la inconsistencia de estados entre ese archivo y los story files individuales persiste como deuda de proceso.
