## Desafíos y Patrones Detectados

### Patrón 1: AC violations en límites sin test unitario previo (P1 y P2 en 8-3)

**P1**: `renderScoreComponent` no verificaba `score.isExcluded` — mostraba scores en sub-rows aunque la FIBRA estuviera excluida (AC5 violado).
**P2**: `parseCompareTickers` no capaba en `MAX_COMPARE_FIBRAS` al parsear la URL — `/comparar?fibras=A,B,C,D,E` cargaba 5 tickers (AC6 violado).

Ambos habrían sido atrapados con tests unitarios específicos en `comparador-logic.test.ts` antes del primer commit. El módulo existe exactamente para hacer testeable esta lógica. El patrón es el mismo de Épicas 6 y 7: AC de límite/borde se detecta en review, no en dev.

### Patrón 2: Duplicación silenciosa de queries (P3) — reincidencia de patrón de Épica 5

`GetLatestSnapshotPerFibraAsync` y `GetSummaryLatestAsync` se llamaban dos veces por request — una para las FIBRAs seleccionadas y otra para el universo del score. El fix fue trivial: reusar los datos del universo filtrando por IDs. El endpoint tiene 7-8 queries en total; sin un grafo documentado previo, es fácil perder la pista de qué ya se consultó.

### Patrón 3: Meta description char count — los 3 chars faltantes (P5)

La meta description tenía 117 chars; el mínimo del checklist es 120. El checklist decía "120-160 chars" pero sin instrucción de verificar. Nadie cuenta caracteres mentalmente. La solución es un hint en el checklist, no mejor intención.

### Bonus: Pre-existing TypeScript build failure

Al resolver D1-D4 se descubrió que el build TypeScript ya fallaba desde la implementación original de 8-3: el schema genera `string | number | null` para campos decimales de `ComparadorScoreDto`, `ComparadorMercadoDto` y `ComparadorDistribucionesDto` donde las funciones helper esperaban `number | null`. Resuelto actualizando las firmas de `formatCompareNumber`, `formatComparePercent`, `formatCompareVolume`, `formatMoney`, `renderScoreComponent` y `scoreToneClass` para aceptar `string | number | null | undefined` con conversión explícita via `Number()`.
